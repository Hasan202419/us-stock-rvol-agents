Texnik xulosa
Mazkur loyihada Telegram-bot yordamida fond bozorini kuzatuvchi va skrining qiluvchi tizim yaratish
bo‘yicha batafsil reja taqdim etiladi. Arxitektura variantlari (serverless, VPS, konteyner), Telegram
integratsiyasi (webhook va long-polling), ChatGPT API ishlatilishi (promptni tuzish, limitlar, xarajatni tejash,
striming), tashqi bozor ma’lumotlari API’lari (jahon va O‘zbekiston bozorlariga oid provayderlar, real-vaqt vs
kechiktirilgan, WebSocket vs REST, autentifikatsiya), skrinner dizayni (indikatorlar, vaqt oralig‘lari, backtest
qo‘llab-quvvatlash, foydalanuvchi qoidalari), avtomatik yangilanish va rejalashtirish (cron, vazifa navbatlari,
hodisa-asosida triggerlar), signal yaratish logikasi (qoidalar misollari, xavf filtrlari, pozitsiya hajmi,
“cooldown” mexanizmi), ChatGPT, bozor API va broker API’larini birlashmasi (multi-API orkestratsiyasi),
xavfsizlik (API kalitlari, foydalanuvchi autentifikatsiyasi, limitlar, ma’lumotlar maxfiyligi), joylashtirish va
monitoring, xarajatlar kabi masalalar yoritiladi. Zarur komponentlar, ma’lumot oqim diagrammasi
(Mermaid), amalga oshirish bosqichlari va jadvali, test rejası, shuningdek 3 ta ma’lumot provayderi va 3 ta
hosting varianti taqqoslanadi. Rasmiy hujjatlar va ishonchli manbalarga tayangan holda batafsil tahlil
beriladi.
Arxitektura variantlari
Serverless (Function): Telegram webhook orqali voqealar kelganda faollashadi; masalan, AWS
Lambda yoki Cloudflare Workers. Ishlatilmagan vaqtda resurs sarfi deyarli yo‘q. Bu model burilishli
yoki kam trafikli botlar uchun iqtisodiy, chunki har bir so‘rov bo‘yicha to‘lashadi
. Masalan,
Cloudflare Workers’da “cold start” juda kichik (<5ms) bo‘lib, globallashgan tarmoq orqali ishlaydi
. Ammo yuqori trafikli doimiy xizmatlar uchun doimiy o‘rnatilgan muhit (kontener yoki VPS)
arzonroq bo‘lishi mumkin. 
Konteyner (Docker/Kubernetes): Botni konteynerga joylashtirib, AWS ECS, GCP Cloud Run yoki
Kubernetes klasterda ishlatish mumkin. Bu arxitektura barqaror ishlash tezligi beradi va murakkab
bog‘liqliklarni boshqarish imkonini beradi, ammo infratuzilma o‘rnatish va boshqarish murakkabroq
. Uzluksiz trafik va aniq kechikish talab qilinsa konteynerlar ustun. 
VPS (Virtual Server): Masalan, DigitalOcean Droplet yoki AWS EC2-da botni doimiy jo‘ylashtirish.
Serverga to‘liq nazorat va istalgan modulni o‘rnatish mumkin. Biroq, u doim yoqilgan bo‘lib, har oy 5–
10$ (yoki ko‘proq) doimiy xarajat talab qiladi. Katta hajmli so‘rovlar uchun resurslar yanada
balanslanishi mumkin, ammo bo‘sh vaqtlar uchun ham to‘lov bo‘ladi. 
Resurslar: Docker, Kubernetes, AWS Lambda/Azure Functions hujjatlari; serverless vs container maqolalari
.

ChatGPT API ishlatilishi
Prompt dizayni: Bot foydalanuvchi so‘rovini ChatGPT modeliga yuboradi. Sokin va aniq
instruktsiyalar bilan samarali prompt tuzing (masalan, “Ishlab chiqarish qoidalariga muvofiq javob
ber” kabi). Keraksiz ma’lumotni qisqartirish va faqat mavzu bo‘yicha savol berish orqali token sarfini
kamaytirish mumkin. Uzun misollarni unchalik yozmaslik kerak; qisqartirish ham token chegarasini,
ham xarajatni kamaytiradi
. Foydalanuvchi kiritmalari tahlili va oldindan tekshiruv ChatGPT
javoblari sifatini oshiradi. 
Rate-limiting va arxitektur jamoasi: OpenAI API so‘rov cheklovlari (requests per minute/token per
minute) mavjud
. Agar bot yuqori tezlikda so‘rov yollasa, 429 xatolik chiqishi mumkin. Bu
holatlarda eksponensial qayta urinuv (exponential backoff) strategiyasidan foydalanish kerak
.
So‘rovlar chegarasiga to‘la yetmaslik uchun max_tokens  parametrini haqiqiy jarayon hajmiga
moslab qo‘yish, promptni soddalashtirish maslahat beriladi
. Shuningdek, turli model (GPT-4,
GPT-3.5) limiti va narxlari farqliligini yodda tutish lozim. 
Striming vs blokli: ChatGPT API’dan striming rejimi bilan foydalanish mumkin (so‘rovga 
stream=true  kiritiladi). Bu bot foydalanuvchi interfeysini tezlatadi — javob bo‘laklari olingan
zahoti ekranga chiqariladi
. Narx tomoni bilan striming yoki blokli formasida foydalanishdagi farqi
yo‘q (hammasi token asosida) boʻlsa-da, foydalanuvchi uchun tezlik sezilarli. Strimingdan foydalanish
uchun Python-da for  sikli orqali kelayotgan deltalarga quloq tutish mumkin
. 
Ollanadigan limitlar: OpenAI tomonidan joriy API uchun 60,000 so‘rov/min (yoki 60 mln token/min)
kabi umumiy limitlar bor; lekin ular zich davr bo‘yicha kvantlash qilinadi (masalan, daqiqada 60k ni
1000 so‘rov/sekund tarzida tatbiq etish)
. Katta token so‘rovlar yoki sukunatli burstlarni oldini
olish kerak. Kelajakda kerak bo‘lsa, yuqori tier (usage tier) ga o‘tish mumkin. 
Manbalar: OpenAI rasmiy hujjatlari va yordam bo‘limi
; ChatGPT striming dokumentatsiyasi
.

Skrinner dizayni va indikatorlar
Indicatoryalar: Skrinner uchun mashhur texnik indikatorlar (SMA, EMA, RSI, MACD, Bollinger Bands,
Stochastic, ADX va h.k.) qo‘llaniladi
. Masalan, ta-lib  kutubxonasi Python uchun 200 dan ortiq
indikatorni (ADX, MACD, RSI, Stochastic, Bollinger Bands, va hokazo) taqdim etadi
. Bu algoritmlar
orqali kurslar pozitsiyasi va o‘zgarishlar boshqariladi. 
Vaqt oralig‘lari: Turli grafiglarda (soatlik, 4-soatlik, kunlik, haftalik) indikatorlar hisoblanadi. Soatlik
yoki minutli (1M, 5M) ma’lumotlar real vaqt skrinning uchun foydali. Bundan tashqari, uzoq muddatli
trendni aniqlash uchun kunlik kurslar ishlatiladi. 
Foydalanuvchi qoidalari: Foydalanuvchilar skrinner qoidalarini o‘zlari belgilashi mumkin (masalan,
“agar SMA20 > SMA50 bo‘lsa va RSI <30 bo‘lsa, sotib olish signali ber”). Qoidalar konfiguratsiya
faylida saqlanib, kerak bo‘lganda qo‘lda yoki bot orqali yangilanishi mumkin. 
Backtesting qo‘llab-quvvatlash: Yaratuvchilar oldindan belgilangan qoidalar va tarixiy ma’lumotlar
asosida strategiyani backtest qilishi kerak. Pythonda Backtrader , zipline  yoki vectorbt
kabi kutubxonalar yordamida strategiya test qilinadi. Bu orqali qoidalar ishchanligini va xavfsizligini
baholash mumkin. 
Risk-filtrlar va balans: Skrinner signallarini tozalash uchun qo‘shimcha shartlar (masalan, bozor
likvidligi, volatillik kx, yangilik koeffitsienti) ishlatiladi. Bu pozitsiya hajmini belgilash va torkorning
(drawdown) oldini olishga yordam beradi.

Avtomatik yangilanish va jadvalga qo‘yish
Cron va rejalashtirish: Agar botga muntazam yangilanish (masalan, har kuni grafikni yangilash)
kerak bo‘lsa, cron-jobs (Linux crontab ) yoki Node-da node-cron  kabi paketlar yordamida vaqtga
asoslangan vazifalarni sozlash mumkin. Misol: har kuni bozor yopilgandan so‘ng skrinnerni ishga
tushirib, keyingi kun uchun signal yaratish. 
Vazifa navbatlari va asinxronlik: Katta ma’lumotlar oqimini boshqarish uchun xabar navbati
(RabbitMQ, Redis Queue yoki AWS SQS) ishlatiladi. Masalan, bozor API’dan kelgan ma’lumotni ilk
tekshiruv uchun navbatga yuborish, fon jarayonida skrinner bloklariga qayta ishlov berish. 
Hodisa-asosida triggerlar: Real-vaqt kurslar (websocket orqali) kelgan zahoti signallarni qayta
hisoblash. Masalan, fini vaqti o‘zgargan qiymat bo‘yicha avtomatik bildirish jo‘natish uchun
WebSocket hodisasi trigger bo‘lishi mumkin. yoki birja yangiliklari chiqqan zahoti stavkalarni qayta
baholash. Bu tarzda "event-driven" muhitda ishlash foydali. 
Manbalar: Texnik bloglar (Node cron masalalari); Telegram botlar bo‘yicha forumlar; AWS SQS/RabbitMQ
dokumentatsiyasi. 
Signal yaratish va xavf boshqaruvi
Signal qoidalari: Misol uchun, “agar oxirgi 10 kunning o‘rtacha narxi (SMA10) 30 kunlik o‘rtacha
narxdan baland bo‘lsa va RSI past (masalan, <30) bo‘lsa, sotib oling” kabi qoidalar tuziladi. Yoki
“macd.sigoal kesib o‘tsin” kabi indikator signalari. Shu kabi bir nechta qatlamli filtrelar orqali soxta
signallar kamaytiriladi. 
Pozitsiya hajmi va risk-filtrlar: Har bir signal bilan birga tavsiya etilgan savdo hajmi hisoblanadi
(masalan, portfelning 1–2% gacha). Masalan, Kelly formula yoki statik foiz usuli (masalan, 1% portfel)
qo‘llanilishi mumkin. Noaniqlik va bozor shovqinini kamaytirish uchun signal peshinini tekshirish
(masalan, oldingi signalga nisbatan salbiy hajm bo‘lsa bekor qilish) yoki issiq boshining ("cooldown")
qoidasini (bir xil aktivga ortiqcha signallar orasida ma’lum vaqtdan keyin ruxsat) kiritish zarur. 
Namuna: Masalan, “Sotib olish signal: SMA(20)>SMA(50) VA RSI<30 VA Ajanlik hajmi ko‘rsatkichi (ADX) >
25”. “Sell signal: narx SMA(20) ostiga tushganda, yoki 50% foydani yig‘gach.” Bunday qoidalar code
modulga kiritilib, har signal paytida tekshiriladi. 
Manbalar: Algoritmik savdo bo‘yicha maqolalar; risolalar (backtesting va signallar haqida).
Ko‘p API orkestratsiyasi
Bot bir nechta tashqi servislarga ulanadi: Telegram API, ChatGPT API, bozor ma’lumotlari API, broker API.
Masalan, foydalanuvchi buyruq yuborsa, avval ChatGPT API orqali so‘zma-so‘z tushuniladi, so‘ng bozor
API’idan kerakli simbol ma’lumotlari olinadi, undan keyin skrinning qoidalari qo‘llanadi va natija broker
API’iga (buyurtma berish uchun) va qayta Telegramga javob sifatida jo‘natiladi. Bunda hamma so‘rovlarni
asinxon (async) usulda bajarish kerak; Node.js yoki Python async kutubxonalari yordam beradi. Har bir
tashqi chaqiriq (ChatGPT, bozor, broker) maxsus modulda (service layer) qo‘llanadi. Turli API chaqiriqlari bir-
biriga bog‘lanmasligi uchun qo‘shimcha state bazasi (masalan, Redis) yoki session ID’dan foydalanish lozim.

Xavfsizlik va maxfiylik
API kalitlarini saqlash: Sirli kalitlarni (Telegram tokeni, OpenAI API kaliti, bozor API kaliti) hech
qachon kod ichiga bevosita yozmaslik kerak
. Ularni atrof-muhit o‘zgaruvchilari yoki maxfiylik
boshqaruv xizmatida (AWS Secrets Manager, Kubernetes Secrets va h.k.) saqlash yaxshi amaliyotdir
. Maksimal kam huquq (principle of least privilege) tamoyiliga amal qilib, har bir kalit faqat
kerakli maqsad uchun ruxsat berilishi lozim. Kalitlarni va’dadan chiqishsiz aylantirib turish (rotatsiya)
kerak. 
Foydalanuvchi autentifikatsiyasi: Bot kerak bo‘lsa, login/auth tizimi qo‘shilishi mumkin; masalan,
foydalanuvchi Telegram ID asosida ruxsat tekshiruvi. Privacy uchun FCM (Flood Control modul) lar
ornatib, botga faqat registratsiyadan o‘tgan foydalanuvchilar kirishini ta’minlash lozim. 
Rate limiting: ChatGPT va bozor API’ga so‘rovlar soni cheklangan. Botda o‘z ichida ham bot
serveriga qarshi DoS hujumlari oldini olish uchun IP bo‘yicha cheklov o‘rnatish va spam-proteksiya
qilish zarur. Misol uchun, bir foydalanuvchi bir daqiqada ko‘p xabar jo‘natishi muammoga sabab
bo‘lishi mumkin, shuning uchun masalan Redis orqali vaqtinchalik ana shu foydalanuvchining
so‘rovlar sonini cheklash mumkin. 
Ma’lumotlar maxfiyligi: Foydalanuvchilarning buyruqlari va javoblari (ayniqsa, moliyaviy maslahat)
shifrlanishini ta’minlash uchun HTTPS kerak. Bot server loglariga foydalanuvchi shaxsiy
ma’lumotlarini tushirmaslik (token, maxfiy so‘rov matni va h.k.) talab qilinadi. 
Manbalar: Stripe xavfsizlik qo‘llanmalaridan xulosa
; umumiy API xavfsizlik amaliyotlari. 
Joylashtirish, monitoring va xarajatlar
Joylashtirish: Tanlangan hostingga kodni CI/CD bilan joylashtirish. Misol: GitHub Actions + Docker +
AWS (ECR+ECS yoki Lambda) yoki Doctl/DigitalOcean CLI. Yagona kontener (Docker Compose)dan
kubernetes (EKS/GKE)gacha variantlar. Har xil muhit (dev, stage, prod) o‘zgaruvchilar bilan
boshqariladi. Flask yoki Node botlarini pm2  yoki gunicorn  yordamida ishga tushurish mumkin.
HTTPS sertifikatlarini avtomatik yangilash (Let’s Encrypt) lozim. 
Monitoring: Bot ish faoliyatini kuzatish uchun log va metrikalar tizimi o‘rnatish – masalan,
Prometheus+Grafana, yoki Papertrail/Sentry kabi xizmatlar. Metrikalar: so‘rovlar soni, javob berish
vaqti, xatoliklar (5xx kodlari). Alert qoida sifatida ChatGPT API limitiga yetganda/bozor ma’lumotlari
offline bo‘lganda ogohlantirish e’lon qilishi kerak.

ishlatilgan tokenlar (bufer + javob) miqdori bo‘yicha to‘lanadi. Tejash uchun kerakli joyda faqat kerakli
model va minimal tokendan foydalanish zarur.

Zarur komponentlar va ma’lumot oqim diagrammasi
Loyihada quyidagi komponentlar bo‘ladi: Telegram Bot (HTTPS webhook), ChatGPT API moduli, Bozor
ma’lumotlari API moduli, broker API moduli, ma’lumotlar bazasi (signal va sozlamalar uchun), jadval/
reklama (cron yoki navbat) xizmatlari. Ma’lumot oqimi quyidagicha: foydalanuvchi Telegram’da xabar
jo‘natadi → Telegram server bot endpoint’iga webhooks orqali yuboradi → Bot server xabarni qabul qilib,
tegishli bo‘limlarga (ChatGPT, bozor API, broker API) so‘rov qiladi → olgan javoblarni birlashtirib, signal
hisoblaydi → natijani Telegram orqali foydalanuvchiga qaytaradi. Pastdagi Mermaid diagrammada bu
jarayon tasvirlangan: 
flowchart LR
    U[Telegram foydalanuvchisi] -->|Xabar yuboradi| TB[Telegram Bot API]
    TB -->|Webhook so‘rovi| Srv[Bot server (Docker/AWS Lambda)]
    Srv -->|so‘rov: ChatGPT API| GPT
    Srv -->|so‘rov: Bozor API| MD
    Srv -->|so‘rov: Broker API| BR
    GPT --> Srv
    MD --> Srv
    BR --> Srv
    Srv -->|signal saqlash| DB[(Ma’lumotlar bazasi)]
    Srv -->|javob yuborish| TB
Bu yerda ChatGPT API dan foydalanuvchi so‘rovini qayta ishlash, bozor API’dan narx ma’lumotlari olish,
broker API orqali bozor buyurtmalari berish ketma-ketligi ko‘rsatilgan. DB da signal va foydalanuvchi
qoidalari saqlanadi. 
Amalga oshirish bosqichlari (timeline)
Loyihani bosqichma-bosqich amalga oshirish uchun quyidagi tartib tavsiya etiladi:
1. Talablar aniqlash va reja tuzish: Arxitektura variantlarini tanlash, komponentlar ro‘yxatini aniqlash.
2. Muhit sozlash: Git repozitoriy yarating, Docker yoki loyiha muhiti konfiguratsiyasi, zarur kutubxonalarni
o‘rnatish (Python/Node paketlari, OpenAI SDK, ta-lib va h.k.).
3. Telegram webhookni ishlab chiqish: Eng oddiy “Salom Dunyo” botini yaratib, Webhook orqali xabar
oladigan endpoint yozing. Masalan, Flask’da @app.route('/', methods=['POST'])  bilan handler
yozish. (Quyidagi kod misolida Python ishlatilgan.)
4. ChatGPT integratsiyasi: OpenAI API ga ulanishni tashkil qiling. Foydalanuvchi xabarini ChatGPT ga jo‘rib,
olgan javobni qaytaring (socket optionlari bilan). Test qiling: oddiy savollarga ChatGPT javob beradi.
5. Bozor ma’lumotlari moduli: Tanlangan ma’lumot provayderi (masalan, Finnhub) API’dan oddiy so‘rov
qilib, amaldagi narxlarni oling. REST so‘rovni amalga oshirish va JSON parse qiling. WebSocket kerak bo‘lsa,
vidjet bilan ulanishni tashkil qiling (Python’da websocket-client  kabi).
